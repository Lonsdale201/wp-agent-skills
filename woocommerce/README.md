# woocommerce

Skills for **WooCommerce core and the WooCommerce extension family** ‚Äî WC itself, plus official extensions like Subscriptions, Memberships, the Stripe gateway, etc.

Use these when extending Woo or building an integration that touches Woo data.

## WooCommerce core

| Skill | Purpose |
|---|---|
| `wc-shipping-method` | Register a custom shipping method with explicit control over which fields appear in the per-zone settings modal ‚Äî extend `WC_Shipping_Method`, declare fields in `init_form_fields()` (no `unset` / DOM hacks / CSS hides), use `$supports = array( 'shipping-zones' )` to suppress the modal entirely. Corrects the "this is React, removing fields is hard" misconception ‚Äî the zone-method modal is Backbone, field list is fully PHP-controlled. |
| `wc-product-search-select` | Build a WooCommerce-style AJAX product picker (`class="wc-product-search"` + `data-action="woocommerce_json_search_products_and_variations"`) ‚Äî products AND variations through WC's built-in endpoint, server-side pre-render of saved options via `wc_get_product()->get_formatted_name()`, no manual enqueue on WC admin screens (auto-enqueued), explicit enqueue for non-WC pages. Solves the "load all 20k products into a `<select>`" antipattern. |
| `wc-hpos-compatibility` | Make a plugin HPOS-compatible (default-on in WC 10.x) ‚Äî declare via `FeaturesUtil::declare_compatibility` on `before_woocommerce_init`, replace direct `$wpdb->postmeta` / `WP_Query` order code with `wc_get_orders` + `WC_Order::get_meta`, build admin hook names dynamically via `OrderUtil::get_order_admin_screen`. Solves the "my plugin worked on dev but finds no orders on prod" silent breakage. |
| `wc-variations-data` | Read, query, and write WooCommerce variations correctly ‚Äî `WC_Product_Variable` (parent) vs `WC_Product_Variation` (child) class split, the `wc_var_prices_<id>` cache, the right programmatic-creation sequence (`set_parent_id` + `set_attributes` + `save` + `wc_delete_product_transients` + `WC_Product_Variable::sync`), the three-level stock model. Solves "I added a variation programmatically and the parent's price range / stock didn't update." |
| `wc-variations-pricing-filters` | Mutate variation prices via filters ‚Äî pick the right layer (`woocommerce_product_get_price` vs `woocommerce_product_variation_get_price` vs the `woocommerce_variation_prices_*` aggregation family), and CRITICAL: filter `woocommerce_get_variation_prices_hash` whenever your logic depends on context outside the default cache key. Solves both "min/max range doesn't update" and "everyone gets the first user's cached discount" bugs. |
| `wc-variation-gallery` | Read/write/render WooCommerce 10.9+ native variation galleries (the merged Additional Variation Images feature) ‚Äî the experimental `variation_gallery` feature flag, the split storage (primary image in `set_image_id()` / `_thumbnail_id`, extras in `set_gallery_image_ids()` ‚Üí `_product_image_gallery`, primary excluded from the gallery list), REST v3 `gallery_image_ids` writes, the classic `variable.php` gallery-swap + reset-snapshot contract and `wc-product-gallery-before-destroy` events, and safe legacy `_wc_additional_variation_images` migration. Distinct from swatches: galleries set per-variation images. |
| `wc-product-attribute-swatches` | Read/write/render WooCommerce 10.9+ visual attribute swatches ‚Äî the experimental `wc-visual` attribute type (block-theme-gated), the global `pa_*` taxonomy with mutually-exclusive `color` (hex) / `image` (attachment ID) term meta, Store API `__experimental_visual` ‚Üí `__experimentalVisual` opt-in reads, and the classic-theme pattern of driving the native `attribute_pa_*` select from accessible swatch `<button>`s (never replacing it). Corrects "swatches live on variations" ‚Äî in core they're attribute-term data, not `product_variation` meta. |
| `wc-shipping-providers` | Register a custom carrier identity for the WC Fulfillments system (WC 10.1+) ‚Äî extend `AbstractShippingProvider` (4 abstract methods: `get_key` / `get_name` / `get_icon` / `get_tracking_url`), register via `woocommerce_fulfillment_shipping_providers` filter. Distinct from shipping methods ‚Äî providers are post-purchase tracking-aware carrier identities, NOT checkout-time rate calculators. |
| `wc-rest-api-v4` | Use WooCommerce REST API v4 (namespace `wc/v4`, since WC 10.2) ‚Äî verified route catalog, hook prefix `woocommerce_rest_api_v4_<route>_*`, when to pick v4 over v3 (DELETE on shipping zones, fulfillments CRUD, segmented settings, ID-sortable customers), and the rule that the v4 `AbstractController` is `Internal\` (NOT a public extension surface ‚Äî plugin-defined routes still use `WP_REST_Controller`). |
| `wc-payment-gateway` | Register a custom payment gateway ‚Äî extend `WC_Payment_Gateway`, implement `process_payment` returning `array(result, redirect)`, optional `process_refund`, declare features in `$supports`, register via `woocommerce_payment_gateways`. The `payment_complete` vs `update_status` distinction (canonical paid-order state machine vs status-only flip) and the always-forgotten `WC()->cart->empty_cart()` after success. Webhook receiver via `wc-api`. |
| `wc-emails-classic` | Customize WC transactional emails the classic PHP-template way (NOT block editor) ‚Äî extend `WC_Email`, register via `woocommerce_email_classes`, hook `trigger()` to `_notification` actions, use `wc_get_template_html` with `template_base`. Plus the theme `woocommerce/emails/<file>.php` override path that doesn't need a class. |
| `wc-coupon-dynamic` | Synthesize WC coupons at runtime via `woocommerce_get_shop_coupon_data` ‚Äî no `shop_coupon` posts needed. The hidden virtual-coupon mechanism for rule-driven codes (LOYALTY-{ID}, partner CRM, auto-apply discounts), custom discount types, and the validation triple `woocommerce_coupon_is_valid_*`. The single most AI-deficient WC topic. |
| `wc-customer-and-sessions` | Use WC's session and customer APIs ‚Äî `WC()->session` (cookie + `wp_woocommerce_sessions` table) for visitor data, `WC()->customer` for the active customer context including guests, `new WC_Customer($user_id)` for one-off loads. Replaces the broken `$_SESSION` / `setcookie` / user_meta-for-guests patterns AI defaults to. |
| `wc-order-lifecycle-and-items` | Work safely with order statuses, payment completion, status hooks, order items, line-item meta, totals, and stock side effects ‚Äî `payment_complete()` vs `update_status()`, `woocommerce_order_status_*` hook ordering and args, `WC_Order_Item_Product`, `add_item()`, `calculate_totals()`, stock reduction/restoration hooks, HPOS-safe CRUD, and why not to instantiate base `WC_Order_Item`. Solves "my paid order skipped the lifecycle side effects" and order-item meta confusion. |
| `wc-cart-checkout-classic` | Customize the classic (shortcode) cart and checkout ‚Äî `woocommerce_add_cart_item_data` / `woocommerce_get_item_data` cart-key merging, `woocommerce_before_calculate_totals` absolute price mutation, `woocommerce_cart_calculate_fees`, `woocommerce_checkout_fields`, `woocommerce_after_checkout_validation`, order-line meta vs order meta, HPOS-safe order saves, and the Checkout Block / Store API boundary. Solves missing/duplicated cart and order-item data. |
| `wc-store-api` | Build against the WooCommerce Store API (`/wp-json/wc/store/v1`) for shopper-facing products / cart / checkout, Cart & Checkout Blocks, and headless carts ‚Äî route choice, Nonce header (`wp_create_nonce('wc_store_api')`) vs Cart-Token, sessions / CORS / rate limits, `woocommerce_store_api_register_endpoint_data`, `/cart/extensions` + `extensionCartUpdate`, payment requirements, the `related` product-query pitfall, and when to use WC REST `wc/v4` instead. | `wc-payment-tokens` | Store and use saved payment methods through `WC_Payment_Tokens` / `WC_Payment_Token_CC` ‚Äî provider token vs raw card data, tokenization gateway support, create/update/delete/default, My Account nonce + ownership checks, `get_customer_tokens` / `get_customer_default_token` / `get_order_tokens`, attaching tokens to orders, gateway ID filtering, custom token tables, HPOS-safe order use, and checkout saved-token validation. | `wc-admin-inline-scripts` | Add small WooCommerce admin JavaScript the modern way ‚Äî replace the deprecated (since WC 10.4) `wc_enqueue_js()` / `wc_print_js()` / `woocommerce_queued_js` global footer queue with a registered handle + `wp_add_inline_script()`, gate enqueues by `$hook_suffix` / screen / settings tab, preserve the old jQuery-ready wrapping that `wp_add_inline_script` does NOT add, pass PHP data safely with `wp_json_encode()`, and attach to Woo admin handles (`woocommerce_admin`, `wc-enhanced-select`, `wc-shipping-zones`). |
| `wc-action-scheduler-jobs` | Queue and run WC background jobs with Action Scheduler ‚Äî `as_enqueue_async_action`, `as_schedule_single_action`, `as_schedule_recurring_action`, `as_schedule_cron_action`, exact-arg checks (`as_has_scheduled_action`), the delivery contract (due is not run; no exactly-once / FIFO / max start time), remote idempotency keys instead of local success markers, bounded retries with backoff + `action_scheduler_failed_execution` telemetry, the AS 3.9.3 daily `action_scheduler_ensure_recurring_actions` repair hook behind `as_supports()`, the consistently-failing recurring pause (default threshold 5), groups as namespaces (NOT locks/FIFO), JSON args passed via `array_values()`, WP-CLI diagnostics (`version` / `source` / `data-store` / `runner` / `status` + `run` / `action`), batching, and the WC 10.9.4 DB-store gotcha that `$unique` keys on hook + group (NOT per argument set). | `wc-logging` | Add production-safe WooCommerce logging with `wc_get_logger()` ‚Äî stable log sources, severity levels and thresholds, structured `context`, correlation IDs, sensitive-data redaction, the DB vs file handler + retention model, volume control, custom handlers, and why logs are diagnostics not durable business state.

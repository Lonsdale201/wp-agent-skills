# lw-plugins

Skills covering the **LW Plugins family** (LW LMS, LW Site Manager, El├бll├бs for WooCommerce, LW Firewall). Use these when extending or building on top of these plugins.

## Skills

| Skill | Purpose |
|---|---|
| `lw-lms-backend-extend` | Backend extension contract for `lwplugins/lw-lms` (BETA тАФ README explicitly says "not recommended for production use"), verified for v1.6.0 / PHP 8.2+. Headless LMS тАФ courses / lessons / sections / progress / access control. v1.6.0 makes `lw_lms_has_course_access` the final logged-in paid-course decision after built-ins and adds `AccessRepository::revoke_by_source()` for integration-owned stored grants. Covers access/progress hooks and repositories, source/source-ID caveats, settings tabs, WooCommerce subscriptions/memberships, 11 custom capabilities, three DB tables, and optional Site Manager integration. Routes abilities, frontend, migration, and WP-CLI topics to sibling skills. |
| `lw-lms-wp-cli-operations` | Operational LW LMS WP-CLI commands added in v1.4.0, verified against v1.6.0 тАФ `wp lw-lms course create\|list\|delete\|set-section`, `lesson create\|list\|assign`, `enroll`, `revoke`, `force-complete`. Covers argument resolution, repository/hook side effects, and CLI footguns. The v1.6.0 `revoke_by_source()` API is PHP-only: CLI `revoke` still changes the first active row regardless of source and exposes no source selector. **Not** the LearnDash migration command. |
| `lw-lms-rest-frontend` | Build a custom frontend against `lwplugins/lw-lms`'s headless `/wp-json/lms/v1` REST API (BETA, verified for v1.6.0). The route/schema catalog is unchanged; v1.6.0 server-side paid-access filters can now affect `access.has_access`, lessons, progress writes, attachments, and downloads. Since v1.4.0 single-course `content` is PUBLIC marketing/about content; only lesson content stays access-gated. Denied paid-course `access` can carry products, subscriptions, subscription variations, and WooCommerce Memberships plans. |
| `lw-lms-abilities` | Consumer / reviewer reference for the built-in `lw-lms/*` Abilities API surface registered by lw-lms v1.6.0 тАФ `list-courses`, `get-course`, `get-progress`, `set-progress`, `get-options`. The v1.6.0 access changes added no ability. Covers the Site Manager bridge, standalone WP 6.9+ fallback, permissions, `input`-wrapped REST calls, and output schemas. |
| `lw-lms-learndash-migration` | Plan, run, or review the one-time WP-CLI LearnDash тЖТ LW LMS migration (`wp lw-lms migrate-learndash --dry-run --verbose`) in lw-lms v1.6.0; the v1.6.0 access changes did not alter its flow. Maps `sfwd-courses` / `sfwd-lessons` into `course` / `lesson` posts, writes reverse mappings, builds generated sections from `ld_course_steps`, and extracts video URLs. Rerunnable but not a sync; it does not migrate quizzes, certificates, groups, assignments, user progress, enrollments, or drip rules. |
| `lw-site-manager-overview` | Reference for `lwplugins/lw-site-manager` тАФ a WP 6.9+ Abilities-API-native exposure layer that registers 120+ machine-callable abilities under `site-manager/*` for AI agents (Claude, ChatGPT, MCP clients) to discover and invoke. Calling pattern via `/wp-json/wp-abilities/v1/abilities/{namespace}/{ability}/run` with Application Password Basic auth. **Not** MainWP тАФ different surface and security model (per-ability cap-checks). |
| `lw-site-manager-extend-abilities` | Add custom abilities to LW Site Manager via two action hooks (`lw_site_manager_register_categories`, `lw_site_manager_register_abilities` тАФ second receives the central `PermissionManager` instance). Critical pattern тАФ extend `AbstractAbilitiesRegistrar` to inherit the meta builders (`readOnlyMeta` / `writeMeta` / `destructiveMeta`) and schema builders (`paginationSchema` / `orderingSchema` / `idSchema` / `listOutputSchema` / etc.). |
| `lw-elallas-integration` | Integrate with or extend **El├бll├бs for WooCommerce** (`elallas-for-woo`, verified v1.0.12) тАФ the EU/HU right-of-withdrawal (el├бll├бs) case-management plugin. An LW-family plugin (namespace `LightweightPlugins\Elallas`, `wp_lw_elallas_*` tables) you integrate by CONSUMING hooks, never editing. Covers the eligibility/deadline/B2B/order-number filters (`elallas_is_order_eligible`, `elallas_deadline_days`, `elallas_is_order_b2b`, `elallas_delivery_date`, and the new `elallas_resolve_order_number` for custom/sequential order numbers тАФ never `absint()` the display value), the case-lifecycle actions (`elallas_case_created` / `_confirmed`, the now **4-arg** `elallas_case_status_changed` with the customer-visible note, `elallas_invoicing_case_created`), `elallas_pdf_html` and `elallas_boot`, the `CaseStatus` lifecycle, product/category/tag withdrawal exceptions (`_lw_elallas_excluded` тЖТ `eligibility_flag = 'excepted'`), the runtime WPML/Polylang string + page-ID translation paths (and the intentionally narrow `wpml-config.xml` scope), PII-scrubbed WooCommerce logging, the 4 custom tables + read-only order meta, and the `elallas/*` LW Site Manager abilities. HPOS-safe; detect with `defined('ELALLAS_FOR_WOO_VERSION')`. |
| `lw-firewall-rate-limit-worker` | Make your plugin compatible with **LW Firewall** (`lw-firewall`, verified v1.3.2) and its MU-plugin worker that runs on `muplugins_loaded` priority 1. Covers the fixed worker check order (kill switch тЖТ `enabled` тЖТ server/localhost skip тЖТ `ip_whitelist` тЖТ `ip_blacklist` тЖТ geo тЖТ auto-ban тЖТ 404 flood тЖТ bot UA тЖТ endpoint detection тЖТ rate limit), which request types the worker actually detects (`cron`/`xmlrpc`/`login`/`rest`/`filter` тАФ arbitrary pretty URLs are NOT rate-limited), narrow `filter_params` substring rules with per-prefix limits, endpoint-local self-protection via `RateLimiter::is_allowed_key()` + `AutoBanner` + `lw_firewall_resolve_storage()`, the `IpDetector::get_ip()` Cloudflare-range trust model, whitelist/blacklist CIDR handling, worker lifecycle (install/remove/version-mismatch bail), response codes, and the `LW_FIREWALL_*` wp-config overrides. |
| `lw-firewall-registration-guard` | Wire a **custom registration form** into LW Firewall's spam protection (v1.3.2). The plugin auto-protects only the default WP `register_form`/`registration_errors` (when `users_can_register`); custom/AJAX/REST/Woo/CRM/LMS signups must opt in. Covers the proof-of-render token (`lw_fw_reg_token`) + honeypot (`lw_fw_url`), the public methods `RegisterGuard::render_fields()` / `RegisterGuard::validate( WP_Error )` for server-rendered forms, headless token issue via `RegisterToken::issue()`, manual validation with `RegisterToken::verify( $token, $min_fill_time, $max_age, $storage )`, single-use replay protection, generic error messaging (never leak which check failed), and reject-count auto-ban shared with the worker store. Reminds you the token is anti-spam, not a WordPress nonce тАФ keep your own CSRF/cap checks. |
| `lw-firewall-management-abilities` | Automate or audit **LW Firewall** management (v1.3.2) across its three surfaces тАФ WP-CLI (`wp lw-firewall status\|config\|config-items\|ip\|worker`), LW Site Manager abilities (`lw-firewall/get-options`, `get-log`, `list-blocked`, `block-ip`, `unblock-ip`, all `can_manage_options`), and admin import/export. Covers the single-option model (`Options::get_all()` / `save()` merge-with-defaults, list-typed keys), `LW_FIREWALL_*` constants winning over saved options for single-value reads, why abilities only touch the **manual** `ip_blacklist` (auto-bans are stored separately), the capped `lw_firewall_log`, MU-worker status/reinstall, and safe config migration. |

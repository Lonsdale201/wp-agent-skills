# learndash

Integration and audit skills for **[LearnDash LMS](https://www.learndash.com/)** (`sfwd-lms`) and the **LearnDash WooCommerce** bridge (`learndash-woocommerce`) — the enrollment, group, and headless surfaces a crossover plugin, theme, import, or CRM/checkout bridge has to get right.

These are skills for making **your own** code cooperate with LearnDash, not for re-implementing it. LearnDash already owns the access decision (`sfwd_lms_has_access()`), the user/course/group/leader relationship model, course-step structure, the WooCommerce access counter, and the REST controllers under `ldlms/v2` — a crossover plugin should grant/revoke through the public helpers (`ld_update_course_access()`, `ld_update_group_access()`, `ld_update_course_group_access()`) and read through the access API rather than writing `course_{id}_access_from` / `learndash_group_users_{id}` usermeta by hand. The running theme across the batch: **course access is not one meta key** (it can come from direct enrollment, groups, open courses, sample steps, admin auto-enroll, and product start/end + expiration windows, so authorization goes through the access API), **grant/revoke through the LearnDash helpers** so activity records, historical `enrolled_at` timestamps, transients, and hooks all fire, and **respect the WooCommerce bridge's model** — product/variation `_related_course` / `_related_group` relations, configurable grant/revoke statuses, partial refunds, subscription switches, HPOS-safe order CRUD, the silent enrollment queue, and the per-user access counter that stops one order from revoking access another still grants. Verified against LearnDash LMS 5.1.6.1 and LearnDash WooCommerce 2.0.2 local source.

## Skills

| Skill | Purpose |
|---|---|
| `learndash-course-access` | Course access and enrollment: `sfwd_lms_has_access()` as the canonical gate and its full decision chain (direct/group/open/sample/auto-enroll/free-paynow, start-end window, expiration), granting/revoking through `ld_update_course_access()` and the `Product` model (not raw usermeta), `ld_course_access_from_update()` for historical access dates, the read APIs (`ld_course_access_from`, `ld_course_access_expires_on`, `learndash_user_get_enrolled_courses`, `learndash_get_users_for_course`), the forced-`$bypass_transient` and open-course query caveats, and the course-steps outline APIs (course structure is not access). |
| `learndash-group-access` | Group membership and group-driven course access: the three separate relationships (user → group, group → course, leader → group), the correct helpers (`ld_update_group_access`, `ld_update_course_group_access`, `ld_update_leader_group_access`, the `learndash_set_*` diff helpers), where each meta lives (user meta vs course postmeta), the read APIs (`learndash_is_user_in_group`, `learndash_get_users_group_ids`, `learndash_group_enrolled_courses`, `learndash_user_group_enrolled_to_course`), hierarchical-group expansion that a direct meta scan misses, 5.x transient changes, and why group access can make `sfwd_lms_has_access()` true with no direct course meta. |
| `learndash-woocommerce-access` | The LearnDash WooCommerce bridge that grants/revokes access from Woo products, orders, refunds, subscriptions, switches, and order-item changes: the `_related_course` / `_related_group` product/variation relation model, the configurable grant/revoke `learndash_woocommerce_enrollment_status_settings`, partial-refund and subscription lifecycle handling, HPOS-safe order CRUD, the `learndash_woocommerce_silent_course_enrollment_queue`, and the per-user `_learndash_woocommerce_enrolled_courses_access_counter` that prevents one order/refund from revoking access another still grants. |
| `learndash-rest-api` | REST / headless integrations: the two surfaces (`ldlms/v2` for course/group/user/steps/progress CRUD, `learndash/v1` manifest/OpenAPI discovery with the `Learndash-Experimental-Rest-Api` header), enablement via `LEARNDASH_REST_API_ENABLED` + settings, the default route bases and why not to hard-code them, the nested ID-keyed `/steps` body shape (not arrays), the enrollment/relation endpoints and their permission model (open-course rejection, group-leader limits), and safe extension via a separate namespaced controller with explicit permission callbacks. |

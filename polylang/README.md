# polylang

Compatibility and extension skills for **[Polylang](https://wordpress.org/plugins/polylang/)** (`polylang`), **Polylang Pro**, and **Polylang for WooCommerce** â€” the multilingual layer for WordPress and WooCommerce.

These are skills for making **your own** plugin or classic theme work correctly on a Polylang site, not for re-implementing Polylang. Polylang already owns language detection, the `language` / `term_language` and `post_translations` / `term_translations` private taxonomies, translated slugs, string translation, and the Woo-specific data stores â€” a crossover plugin should guard and call Polylang's public API and hooks rather than reading `$_GET['lang']`, concatenating `/$lang/` URLs, or writing those taxonomies by hand. The running theme across the batch: **guard the public API** (Polylang may be inactive, have no languages, or run in admin/REST/CLI with no current language), **use the model instead of raw DB/meta writes** so caches, validation, and sync hooks run, and **feature-detect the Pro and WooCommerce layers** (`POLYLANG_PRO`, `PLLWC_Data_Store`) because they change behavior â€” REST fields, shared slugs, ACF strategies, per-language SKU checks. Verified against Polylang 3.8.5, Polylang Pro 3.8.5, and Polylang for WooCommerce 2.2.2 on WooCommerce 10.9.1 / WP 7.0.

## Skills

| Skill | Purpose |
|---|---|
| `polylang-language-api` | Use the public language API safely: the load/guard pattern, `pll_current_language` / `pll_default_language` with their field and composite-property behavior, `pll_languages_list`, localized `pll_home_url`, rendered and raw `pll_the_languages` switchers, `pll_is_translated_post_type` / `pll_is_translated_taxonomy` and the early `pll_get_post_types` / `pll_get_taxonomies` opt-in filters, plus admin/REST/CLI caveats and the common mistakes (reading `$_GET['lang']`, hardcoding URL prefixes, assuming a current language always exists). |
| `polylang-object-translations` | Create, read, link, and update translated posts and terms through Polylang's model, never direct DB writes: `pll_get_post` / `pll_get_term` return semantics (integer ID, `0` on miss), `pll_get_*_language`, `pll_get_*_translations`, the 3.7+ `pll_insert_post` / `pll_insert_term` / `pll_update_*` wrappers with inline `translations` groups, `pll_set_*_language` + `pll_save_*_translations` for existing objects, the media translation caveat, a safe import/migration order, and the `pll_save_post` / `pll_save_term` / `pll_translate_media` hooks. |
| `polylang-pro-slugs-sync-acf` | Polylang **Pro 3.8.5** behavior that changes compatibility: translated rewrite slugs via the `pll_translated_slugs` filter and the `wp_loaded` timing rule, shared slugs (never assume a globally-unique `post_name`), the duplicate/sync model and its `pll_copy_post_metas` / `pll_translate_post_meta` / `pll_post_synchronized` hooks, and the ACF Pro integration â€” the 3.7+ "field groups are not translated" model, the five field translation strategies (`ignore` / `copy_once` / `translate` / `translate_once` / `sync`), relationship-ID collectors, and `pll_enable_acf_labels_translation`. Guard everything behind `POLYLANG_PRO`. |
| `polylang-rest-headless` | REST and headless integrations across the three stacks: the `lang` request parameter and default-language fallback, the `pll/v1/languages` endpoint schema, `pll_filtered_rest_routes` for collection routes, the Pro `lang` / `translations` REST fields and their write permission errors, the 3.8 indexed-array format for `pll_rest_api_post_types` / `pll_rest_api_taxonomies`, `pll_rest_request_object_type` for custom controllers, the Pro `pll/v1/translation` and `pll/v1/untranslated-posts` endpoints, and Woo REST product/order/batch language behavior. |
| `polylang-strings-options` | Translate admin-configured copy and option values: `pll_register_string`'s admin-only registration, output with `pll__` / `pll_e` / `pll_esc_html__` / `pll_esc_attr__` and specific-language `pll_translate_string`, the 3.8 five-argument `pll_sanitize_string_translation` filter (with `$previous`), `PLL_Translate_Option` for recursive option-array keys and wildcards, raw-vs-translated option safety during writes, recipient-language emails/background jobs, and what not to register (tokens, per-request strings, generated HTML). |
| `polylang-wc-compatibility` | Build WooCommerce extensions that work on **Polylang for WooCommerce 2.2.2**: the `PLLWC_Data_Store` product/variation/order language stores, the `lang` query behavior (`lang => ''` for all languages), variation and attribute sync, `pllwc_copy_post_metas` / `pllwc_translate_product_meta` / `pllwc_translate_product_prop`, stock/reserved-stock sync, per-language SKU / global-unique-ID checks, cart and Store API language behavior, order language + HPOS query filtering, Woo REST fields and batch create queues, and translated Woo strings/options. |
| `polylang-compatibility-audit` | The audit / verdict companion to the implementation skills above. Decide end to end whether an existing plugin or classic theme is safe on a Polylang / Polylang Pro / Polylang for WooCommerce site and return a verdict (Compatible / Partially compatible / Not Polylang-compatible / Not testable) plus a severity-tagged findings-and-fix report. Walks each surface in order: stack detection (`POLYLANG_VERSION` / `POLYLANG_PRO` / `PLLWC_Data_Store`), static gettext vs admin-entered dynamic strings (register + `pll__()` / `PLL_Translate_Option`), stored object IDs & URLs needing `pll_get_post` / `pll_get_term` / `pll_home_url` (with the `0`-on-miss trap), CPT/taxonomy opt-in timing, model-based creation/import/sync (never write the private language taxonomies), queries & custom tables (no automatic language filtering â€” needs a `lang` column/snapshot), REST/AJAX/headless `lang` propagation, Pro-only assumptions behind `POLYLANG_PRO`, WooCommerce data-store paths, and async output (cron/webhooks/PDF/email/exports) that must pick an explicit language. Defers to the six implementation skills and `wp-i18n-audit` for the fixes. |

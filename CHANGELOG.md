## 2026-06-17

### New domain — `dev-tooling/` (testing & QA tooling)

A new top-level domain for the **testing / quality-assurance pipeline** around a WordPress plugin or theme — PHPUnit, PHP_CodeSniffer + WordPress Coding Standards, and PHPStan, wired through Composer and CI. Distinct from the runtime/scaffold domains: this is the developer tooling that runs *beside* the code. Deliberately WP-specific (the WP test suite, WPCS rulesets, WP PHPStan stubs) and version-sensitive, so each skill pins tested tool versions and prefers a "verify against your installed version" stance. The version-sensitive facts in these skills (versions, package names, commands, config syntax) were verified against canonical sources (WP core handbook, WP-CLI scaffold-command, WPCS/PHPCS/PHPCompatibility repos, `szepeviktor/phpstan-wordpress`, phpstan.org) and adversarially rechecked before publishing.

### New domain — `elementor/` (Elementor)

A new top-level domain for building **[Elementor](https://elementor.com)** addon / companion plugins — the developer-extension pattern (like `jetformbuilder/` and `jet-engine/`), not a compatibility domain. Four skills covering the Dynamic Tags extension surface plus a deprecations audit. The running theme: the Dynamic Tags **infrastructure ships in free Elementor** (`Elementor\Core\DynamicTags\*`) while the editor picker and AJAX query control are **Pro**, so the skills extend the free base classes, feature-detect Pro, and degrade gracefully. Every claim is grounded in source line references against the tested version (Elementor 4.0.7 free / 4.0.4 Pro).

### New skills (Elementor 4.0.7 free / 4.0.4 Pro / WP 7.0)

- **`elementor/elementor-dynamic-tag-register`** — Register a custom Dynamic Tag: the modern `elementor/dynamic_tags/register` action + `$manager->register( new MyTag() )` (an **instance**), with the legacy `elementor/dynamic_tags/register_tags` + `register_tag( $class )` (a **string**) pair flagged as deprecated since 3.5.0; the four required methods (`get_name` / `get_title` / `get_group` / `get_categories`), `register_group()`, bootstrap timing on `elementor/loaded`, extending the **free** `Tag` / `Data_Tag` bases (not the Pro `Pro_Tag`) for portability, and the Pro-feature reality (free Elementor registers zero tags of its own; Pro ships ~34 behind a license check). `plugin: elementor`.
- **`elementor/elementor-dynamic-tag-fields`** — The body of a tag: `Tag` (echoes via `render()`, content type `ui`) vs `Data_Tag` (returns via `get_value()`, content type `plain`); the value-kind declaration via `get_categories()` and the `Module::*_CATEGORY` constants (so the right controls accept the tag); settings via `register_controls()` (not the deprecated `_register_controls()`) with `Controls_Manager` types; and the fallback asymmetry — a `Tag` gets Before / After / Fallback automatically (applied on empty `render()` output), a `Data_Tag` gets none and must register a `fallback` control and consult it in `get_value()`. Has `reference.md` (full control-type catalog + complete worked `Tag` / `Data_Tag`). `plugin: elementor`.
- **`elementor/elementor-dynamic-tag-ajax-select`** — Pick one item from a large dataset (products / posts / terms / users) without freezing the editor: why a preloaded `SELECT2` hangs the panel at scale, the Pro AJAX query control (`QueryControlModule::QUERY_CONTROL_ID` + `autocomplete` with `QUERY_OBJECT_POST|TAX|AUTHOR|USER|ATTACHMENT`, search-scoped server-side so catalog size is irrelevant), using it inside a Dynamic Tag, that you write **no** `wp_ajax_` handler (Pro owns it), and the required `class_exists( QueryControlModule::class )` feature-detect that degrades to a manual ID `TEXT` field. `plugin: elementor-pro`.
- **`elementor/elementor-deprecations`** — Audit addon code for deprecated Elementor APIs and deprecate your own correctly: the central `Deprecation` class (3.1+) and its six methods (each carrying name + version + replacement, so the list is **source-greppable**), the underscore→no-underscore method renames (`_register_controls` → `register_controls`, `_content_template` → `content_template`, …), the 3.5.0 registration-hook renames (`*_registered` / `register_tags` / `finder/categories/init` → `*/register`), and the debugging gotcha that Elementor's PHP `_deprecated_*` notices need `WP_DEBUG` **and** `ELEMENTOR_DEBUG` within a 4-major window (`SOFT=4` / `HARD=8`). Has `reference.md` (extraction recipe + per-version snapshot). `plugin: elementor`.

### New domain — `translatepress/` (TranslatePress)

A new top-level domain for the [TranslatePress](https://translatepress.com) multilingual plugin, following the same one-domain-per-third-party-plugin pattern as `wp-rocket/`, `redis-object-cache/`, and `fluentcrm/`. These are **compatibility** skills — they make your own plugin/theme behave correctly when TranslatePress is active, rather than extending TranslatePress itself. The domain spans the free core **TranslatePress Multilingual** (`translatepress-multilingual`) and the paid **TranslatePress Business** add-ons (SEO Pack, Multiple Domains, Navigation Based on Language, Translator Accounts, Browse as User Role, Automatic User Language Detection).

### New skills (TranslatePress Multilingual 3.2.1 + Business 1.8.2 / WP 7.0)

- **`translatepress/translatepress-output-compatibility`** — Build plugin/theme output so TranslatePress can translate it: render real server-side front-end text (not opaque data attributes), keep dynamic JS markup stable instead of replacing app roots, and exclude only values that must stay literal via the `data-no-translation` / `data-no-auto-translation` / `data-no-dynamic-translation` / `data-no-translation-href` attributes and the `trp_no_translate_selectors` / `trp_no_auto_translate_selectors` / `trp_skip_selectors_from_dynamic_translation` filters (with the smallest selector, never `.content` / `main`). Covers gettext discoverability, never building a parallel per-language store for ordinary strings, not translating nonces/tokens/IDs/SKUs/API responses, language-aware (and host-aware under Different Domain per Language) HTML cache keys, AJAX/REST fragment language context, the Automatic Language Detection cookie/popup interaction, and the rule that TranslatePress language state is presentation — not an authorization boundary. Cross-links the three sibling TranslatePress skills and `wp-i18n-audit`. `plugin: translatepress-multilingual + translatepress-business`.
- **`translatepress/translatepress-url-seo-compatibility`** — Keep URLs, routing, slugs, and SEO output correct under TranslatePress core URL rewriting and the Business SEO Pack / Multiple Domains add-ons: use WordPress URL APIs and the documented `url_converter->get_url_for_language()` instead of manual `/en/` prefixing or slug concatenation; register public CPT/taxonomy rewrites deterministically (SEO Pack owns translated slugs, and skips uniqueness changes for non-public types); avoid raw `$_SERVER['REQUEST_URI']` routing (SEO Pack temporarily rewrites it on `plugins_loaded` priority 3); steer custom endpoints with `trp_is_rest_api` / `trp_is_admin_link` / `trp_is_form_for_request_uri` / `trp_allow_redirect_to_translated_url`; handle translated WooCommerce `product` / `product-category` / `product-tag` permalinks; mark never-rewrite external links with `data-no-translation-href`; account for Different-Domain host filtering and disabled sitemap caching; and never cache absolute URLs, canonical, `og:url`, sitemap XML, or JSON-LD across languages/mapped domains. Cross-links the sibling skills and `wc-store-api`. `plugin: translatepress-multilingual + translatepress-business`.
- **`translatepress/translatepress-language-ui-navigation`** — Build TranslatePress-aware language UI and navigation: prefer built-in switchers, then `trp_custom_language_switcher()` with a `data-no-translation` wrapper and `hreflang` (never hand-built switcher URLs, which break subdirectory mode / unpublished languages / translated slugs / Different Domain per Language); use the current core `[language-include]` / `[language-exclude]` conditional shortcodes with `enable_translation`, treating legacy `[trp_language]` as deprecated; render menus through `wp_nav_menu()` / `wp_get_nav_menu_items()` so the Navigation Based on Language add-on can hide language-limited items (`_trp_menu_languages`) including nested children; preserve Translator Accounts (`translate_strings`) admin-bar/Translate-Site access when hiding the admin bar; avoid irreversible write side effects during Browse as User Role / translation-editor previews (gate behind POST + nonce + capability); and keep the Automatic User Language Detection popup/hello bar uncovered by sticky headers / consent banners. Cross-links the sibling skills and `wp-accessibility-audit`. `plugin: translatepress-multilingual + translatepress-business`.
- **`translatepress/translatepress-email-notification-compatibility`** — Send plugin / WooCommerce / transactional emails in the recipient's language: prefer `wp_mail()` (core filters subject/body at priority 1 and switches to the first recipient's preferred language — so send separate emails when recipients differ); for custom-rendered templates or non-`wp_mail()` transports, switch with `trp_switch_language()` / `trp_restore_language()` and never leave the language switched, and use `trp_translate()` for isolated strings; for cron / CLI / webhook jobs pick the target language from durable data (`trp_language` user/order meta, default-language fallback) not the current request; preserve WooCommerce order `trp_language` meta via HPOS-safe order APIs (no raw `wp_postmeta` SQL) and correct unusual recipients with `trp_woo_email_language`; allow `[language-include]` / `[language-exclude]` in admin-editable bodies; and never translate names, addresses, order numbers, SKUs, coupon codes, tokens, or payment IDs. Core-side email translation lives in `translatepress-multilingual`; cross-links the sibling skills, `wc-store-api`, and `wp-i18n-audit`. `plugin: translatepress-multilingual`.

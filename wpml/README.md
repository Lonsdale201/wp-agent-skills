# wpml

Compatibility and extension skills for **[WPML](https://wpml.org/)** (`sitepress-multilingual-cms`) — the multilingual layer for WordPress and WooCommerce.

These are skills for making **your own** plugin or theme work correctly on a WPML site, not for re-implementing WPML. The mental model that drives everything: **WPML translates COPIES** — an English page and its Hungarian translation are two distinct posts with their own IDs, linked by a `trid` in `icl_translations` — unlike live-translation plugins that rewrite one page's output on the fly. A crossover plugin should therefore resolve translated IDs at render time, declare its translatable content in `wpml-config.xml`, and register dynamic strings rather than hardcoding cross-language IDs/URLs. The running theme across the batch: **detect the right layer** (base `ICL_SITEPRESS_VERSION` vs the separately-installed String Translation / Media add-ons vs the bundled Translation Management), **prefer declarative `wpml-config.xml`** over runtime code for custom fields / CPTs / taxonomies, and **degrade gracefully** — every hook is a filter/action that no-ops safely when WPML (or an add-on) is absent. Verified against WPML 4.9.5 on WP 7.0.

## Skills

| Skill | Purpose |
|---|---|
| `wpml-overview` | Orient before writing WPML compatibility: the "WPML translates copies" mental model, detecting the base plugin (`ICL_SITEPRESS_VERSION`) vs the add-on split (String Translation `WPML_ST_VERSION`, Media `WPML_MEDIA_VERSION`, bundled Translation Management `WPML_TM_VERSION`) and which feature needs which, the three compatibility mechanisms (declarative `wpml-config.xml`, the runtime language hook API, string registration/translation), a decision matrix mapping intent to mechanism, and where WPML ships its own bundled plugin/theme compatibility. |
| `wpml-config` | Make content translatable declaratively via `wpml-config.xml`: the honored sections (`<custom-fields>` with `translate` / `copy` / `copy-once` / `ignore`, `<custom-term-fields>`, `<custom-fields-texts>`, `<custom-types>` / `<taxonomies>` with `display-as-translated` / `automatic`, `<admin-texts>`, `<shortcode-list>` vs `<shortcodes>`, `<built-with-page-builder>`, `<gutenberg-blocks>`), file discovery + the `wpml_config_array` filter, the exact `0`/`1` and action-enum values, that a typo'd action silently means "ignore", that the XSD is NOT enforced at parse time, and that `<admin-texts>` needs String Translation. |
| `wpml-language-api` | Use the runtime language hook API from plugin/theme code: `wpml_current_language` / `ICL_LANGUAGE_CODE`, `wpml_active_languages`, `wpml_default_language`, `wpml_object_id` to resolve a post/term's translation, `wpml_switch_language` (with the mandatory `null` restore) to run a query in a specific language, `wpml_permalink` / `wpml_home_url` / `wpml_element_link` for language-aware URLs, and `wpml_post_language_details` / `wpml_element_language_details` / `wpml_element_type`. All are filters/actions registered in `SitePress::api_hooks()`; the legacy `icl_*` functions are deprecated since 3.2. |
| `wpml-string-translation` | Register and translate a plugin's dynamic strings (option values, admin-entered labels, runtime-built text — not static gettext): `do_action('wpml_register_single_string', …)` + `apply_filters('wpml_translate_single_string', …)`, the legacy `icl_register_string` / `icl_t` and their `function_exists` guard pattern. Critical: these handlers live in the **String Translation add-on**, not the base — the base only FIRES the hooks, so the hook form degrades safely (no-op / passthrough) while raw `icl_*` calls FATAL without ST. Covers ST detection and when to use `<admin-texts>` instead. |
| `wpml-compatibility-audit` | The end-to-end **audit / verdict** companion to the four implementation skills above. Classify a plugin or classic theme as Compatible / Partially compatible / Not WPML-compatible / Not testable, then walk each surface: static gettext vs dynamic DB/admin strings, stored object IDs that need `wpml_object_id` / `wpml_permalink` resolution, shortcode & block config, queries and custom tables (which WPML does not language-filter), WooCommerce order/product paths (snapshot vs live), and async render paths (emails/PDFs/cron/REST) that do not inherit the request language. Produces a severity-tagged findings + fix-plan report. Cross-references the four implementation skills and `wp-i18n-audit`. |
